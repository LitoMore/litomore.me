Executive Summary

Best valid-path result css-color-converter was fastest in normalizeColor-style mixed badge inputs.
Single SVG badge view Across the tested SVG badge scenarios, the end-to-end gap stayed within roughly 0.21-0.41 microseconds per badge, so rendering work dominates.
Single JSON badge view For JSON badge payloads, css-color-converter saved about 0.54 microseconds per badge, making color normalization a larger share of the work.
Invalid input behavior On invalid inputs, color pays for exception creation/catching while css-color-converter returns null.
Capability difference color supports additional inputs used in the migration, including clamped negative RGB values and hwb(...).

Methodology

  • Each scenario was measured in 7 rounds after 3 warmup rounds.
  • Results are reported using median ops/s and median ns/op to reduce noise.
  • The color benchmark uses the intended migration logic with the obvious startsWith typo corrected so the library is measured fairly.
  • The per-badge section measures end-to-end badge generation with the old and new color implementations injected into the same rendering pipeline.
  • Only overlapping valid inputs were used for the per-badge comparison, and output equality was checked before timing.
  • Inputs are split into overlapping formats both libraries support and new-only formats that color adds.

Interpretation

  • If the workload is mostly end-to-end SVG badge generation, the library choice changes latency only slightly in the tested cases: usually by less than half a microsecond per badge.
  • If the workload is JSON generation or otherwise spends a larger share of time normalizing colors, css-color-converter shows a clearer advantage.
  • If the workload includes many invalid colors, css-color-converter has a structural advantage because it does not use exceptions for control flow.
  • The color migration still buys broader CSS color support, so the performance tradeoff needs to be weighed against that compatibility win.
